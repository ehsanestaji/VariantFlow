\section{Benchmark evidence}

All reported measurements are tied to tracked benchmark reports containing
exact commands, competitor versions, correctness checks, caveats, and
reproducibility commands. VariantFlow compares primarily against bcftools
because it is the trusted performance and correctness baseline for many
VCF/BCF workflows. Rows in Table~\ref{tab:benchmark-summary} and
Figure~\ref{fig:speedup-ranges} are copied from the public release benchmark
table and source reports; no smoke-test row is used as a performance claim.

\vspace{0.8em}

The strongest user-facing evidence is now public FORMAT-rich filtering. On the
DDBJ CHM13 human chr22 cohort with 3715 samples, bounded 1k/10k/50k rows for
DP, GQ, AD, selected-sample DP, and mixed QUAL+FORMAT predicates matched
bcftools core records and measured 4.74--17.78 times faster than
\texttt{bcftools filter}. These rows directly test the selective FORMAT
scanning design on a many-sample human VCF, while still avoiding a default full
27 GB remote cache.

Compressed public-data filtering provides a complementary I/O-heavy result. On
bounded IGSR chr22 public-heavy BGZF inputs, VariantFlow's default native BGZF
path matched bcftools core records and was 13.44--13.47 times faster than
\texttt{bcftools filter}. Conversely, on deterministic FORMAT-heavy stress
data, opt-in predicate parallelism matched bcftools core records and was
4.33--5.27 times faster than \texttt{bcftools filter}; this case exercises the
CPU side of selective FORMAT scanning.

The columnar evidence supports a separate workflow claim. For repeated
analytical queries, VariantFlow exports selected columns to Parquet and then
uses DuckDB for repeated predicates and grouped summaries. In the measured
bounded IGSR workflows, DuckDB results over VariantFlow Parquet matched
normalized bcftools baselines and reduced amortized runtime by 3.18--25.67
times. This does not mean Parquet replaces streaming VCF filters; it means
export-once, query-many workflows can avoid repeated decompression and parsing.
