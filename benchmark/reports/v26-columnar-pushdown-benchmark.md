# VariantFlow v2.6 Columnar Pushdown Benchmark

This report tracks the v3.0 evidence gate for Parquet row-group sizing and
DuckDB repeated-query workflows. It is a workflow-specific claim: export once,
then run repeated analysis queries faster than repeated VCF scans. It is not a
claim that Parquet replaces the native streaming filter.

## Configuration

- Environment: Docker/Linux image `variantflow-v30`
- DuckDB: Python package `duckdb==1.5.2`
- hyperfine: `1.19.0`
- bcftools: `1.21`
- Dataset: deterministic stress VCF generated by the benchmark harness
- Record tiers: `100000 1000000`
- Runs: `3`
- Warmup: `1`
- Repeated DuckDB queries: `5`
- Row-group candidates: `8192 65536 262144`
- Query matrix: `qual_gt_30 dp_gt_40 filter_pass group_by_chrom_filter`

Exact command templates:

- Export: `./target/release/vcf-fast convert <dataset> --to parquet -o <parquet>`
- DuckDB: `/opt/duckdb-venv/bin/python benchmark/query_parquet_duckdb.py <parquet> --query <query> --repeats 5`
- Competitor: `./benchmark/bcftools_columnar_baseline.sh <dataset> <query> 5`

## Claim Decision

All v26 rows matched the normalized bcftools baseline. Query-only DuckDB speedup
ranged from `2.51x` to `240.27x`; amortized export-plus-query speedup ranged
from `1.19x` to `19.20x`. The strongest amortized row in this run was
`INFO/DP > 40` at row group `65536` on the 1M tier (`19.20x`). The weakest
accepted amortized row was grouped `CHROM,FILTER` counts at row group `65536`
on the 100k tier (`1.19x`).

## Caveat

These rows use deterministic stress VCFs. They support the repeated-query
Parquet workflow claim for the measured query shapes, not a replacement claim
for one-off streaming filters or unsupported Parquet schemas.
